## Usage Patterns

### Finding the Right Agent

**By Task Type**:
```bash
# Web development
grep -i "react\|vue\|frontend" agents-catalog.md

# Security tasks
grep -i "security\|audit\|vulnerability" agents-catalog.md

# Database work
grep -i "database\|query\|schema" agents-catalog.md

# Machine learning
grep -i "ml\|machine learning\|model" agents-catalog.md
```

**By Tool Access**:
- **Read-only agents**: Good for analysis, review, advisory
- **Read + Write + Edit**: Development and implementation
- **Read + Write + Edit + Bash**: Full-stack, testing, deployment

**By Model**:
- **haiku**: Fast agents for frequent tasks (review, linting)
- **sonnet**: Balanced agents for most development work
- **opus**: Complex analysis (security, architecture, optimization)

### Installing an Agent

1. Browse this catalog to find relevant specialist
2. Read the agent file from `assets/examples/<domain>/<agent-name>.md`
3. Save to `.claude/agents/<agent-name>.md` (project) or `~/.claude/agents/<agent-name>.md` (global)
4. Customize if needed (tools, model, system prompt)
5. Invoke: "Use the <agent-name> to..."

### Combining Multiple Agents

Create workflow with multiple specialists:
```
"First use the database-architect to design the schema,
then use the backend-architect to design the API,
then use the security-auditor to review the design."
```

### Agent Naming Conventions

All agents follow consistent naming:
- Lowercase with hyphens: `ml-engineer`, `security-auditor`
- Role-based: Describes what they do, not who they are
- Domain-scoped: Specific specialty, not generic "developer"
