# Audit v4: ETUS PMDocs — Elicitation, Coverage, Handoff

**Date:** 2026-03-24
**Framework version:** v5.3
**Base:** Full source read + 4 prior audit iterations + file-level state verification
**Goal:** Define how ETUS must evolve to extract ideas with depth, close semantic gaps in discovery/brainstorm, and deliver a truly implementation-ready handoff to engineering

---

## Central Thesis

ETUS PMDocs already has the right conceptual architecture:

- ideation before specification
- upstream coverage via actors, JTBDs, journeys, use cases, edge cases and assumptions
- traceability chain
- SST
- gates per phase
- workflows per mode

This places the framework above most product documentation approaches.

But the main problem remains:

> **ETUS is still strong in declared methodology and weak in operational enforcement.**

That means:

- the right questions still depend too much on the agent "behaving well"
- coverage still measures quantity far more than semantic completeness
- validation is still more policy than mechanism
- the developer still receives information that is too distributed

The goal of v4 is not "more documents". It is:

1. an interview core genuinely guided by state
2. a discovery more oriented toward real behavior
3. a brainstorm more disciplined and less generic
4. semantic coverage instead of counting
5. executable validation
6. a final handoff packet that does not violate SST
7. change management that actually propagates

---

## Where The Most Important Phases Are

If the question is "where does ETUS most need to be excellent at interviewing?", the answer follows five layers:

### Layer 1 — Core Interview Layer

- `ideate`
- `discover`
- `project-context`
- `product-vision`

This is the most important layer for idea extraction.

It is here that the framework must:

- understand the real problem
- differentiate symptom from cause
- pull concrete stories
- extract constraints and anti-requirements
- identify actors, operations and conflicts
- translate everything into well-framed opportunities

### Layer 2 — Structured Brainstorm Layer

- `brainstorm` within `ideate`
- `solution-discovery`

Second most critical phase, but with a different function:

- not extracting the raw problem
- but transforming evidence into testable directions

If discovery fails, brainstorm becomes guesswork.
If brainstorm fails, the solution becomes premature fixation.

### Layer 3 — Translation Layer

- `feature-brief`
- `prd`
- `user-stories`
- `feature-spec`

Not the primary layer for interviewing, but where discovery and brainstorm quality must survive without degradation.

### Layer 4 — Assurance Layer

- `validate-gate`
- `check-traceability`
- `check-sst`
- new validators

Does not discover the idea, but must prevent ambiguities from escaping.

### Layer 5 — Change Layer

- `correct-course`
- state/handoffs
- post-implementation feedback

Protects the framework from drift after scope changes.

## What Already Works Well (preserve)

### 1. The macro ideation sequence is correct

The order: ingest → problem → actors → jobs → journeys → cases → edges → assumptions → brainstorm → synth is good and must not be discarded.

### 2. The framework already has seeds of anti-requirements

Important signals exist in: Out-of-Scope, constraints and guardrails, questions like "what not to do". The problem is not total absence. The problem is lack of first-class enforcement.

### 3. The traceability infrastructure is already a real advantage

ETUS already thinks better than most frameworks about: upstream/downstream linkage, ownership per document, IDs, handoff between phases. This must be used as the base of the next evolution, not replaced.

### 4. The conceptual validation already exists

There is good intent in: 3-layer validation, adversarial review, pressure test, assumption audit. What is missing is making it more executable.

## Refined Diagnosis

### D1 — The framework is still prompt-rich and enforcement-poor

This remains the best formulation of the problem.

ETUS already contains several good rules: one question per message, section-by-section approval, track unknowns, show progress, pressure test.

But these rules still live primarily in: skill text, agent instructions, templates.

And not in: persistent state, progression blockers, validators, operational artifacts.

**Practical consequence:** An apparently "good" interview can still: skip an essential question type, accept a vague answer, mark coverage without sufficiency, advance due to fatigue or haste.

### D2 — Coverage is still accounting, not semantic

Today the framework counts well: how many actors, how many journeys, how many use cases, how many edge cases.

But the right question for implementation is:

> did we cover the right dimensions for this type of problem?

Examples of dimensions that can still disappear: permissions, side effects, notifications, degraded behavior, data rules, observability, policy and compliance, anti-requirements, concurrency, retry, idempotency.

**Conclusion:** `coverage-matrix` needs to evolve, but without becoming a monolith of responsibilities.

Therefore, v4 proposes separating:

- **coverage** of problem and dimensions
- **interview execution state** of the conversation

### D3 — Discovery and brainstorm need different dynamics

An important improvement over previous versions is making this very clear:

**Discovery must optimize for:** truth, context, real behavior, concrete friction, impact, constraints.

**Brainstorm must optimize for:** breadth of options, structure, clustering, disciplined choice, assumption articulation, test planning.

Today ETUS already suggests this separation, but does not yet execute it with sufficient clarity.

### D4 — Developer handoff is a packaging problem and must not violate SST

Yes, ETUS needs a consolidated artifact for dev.

But v4 adds a critical rule:

> The handoff packet cannot become a new source of truth.

It must be: **derived, generated, regenerable, non-authoritative.**

Otherwise the framework trades one problem for another: before, fragmented information; after, duplication and drift between the packet and the source documents.

**SST rule for the handoff packet:**

Each section points to its authority:

| Packet section | Authoritative source |
|----------------|---------------------|
| Business rules | feature-spec |
| Acceptance criteria | user-stories |
| API contracts | api-spec |
| Data semantics + validation | data-dictionary |
| NFRs | tech-spec |
| Error handling | feature-spec (ES-#) |
| Permissions | feature-spec or tech-spec |
| Observability | quality-checklist |

The packet summarizes and consolidates, but does not redefine.

### D5 — Scope change is the most underdeveloped point of the framework

This remains a serious asymmetry.

The framework is relatively sophisticated for: extracting, structuring, generating, validating.

But still weak for: propagating changes, preserving rejections, remembering user objections, updating connected docs.

### D6 — Edge cases can be identified but never resolved

The traceability chain validates `BO-# → PRD-F-# → US-# → FS-# → impl-#`.

But it does not validate that each `EDGE-#` identified during ideation has a corresponding resolution downstream — a Given/When/Then that handles it, a FS-# rule, an ADR, or an API error code.

An edge case can pass through the entire pipeline undiscovered: identified in ideation, never referenced in any downstream document, invisible to the gate because `failure_modes` dimension was marked `true` based on OTHER edge cases that were handled.

## Recommendations

### R1 — Create two distinct artifacts: `coverage-matrix` and `elicitation-state`

**Problem:** Previous versions pushed too much responsibility onto `coverage-matrix.yaml`, mixing: interview status, answer sufficiency, semantic coverage, blockers, traceability.

**Proposal:**

**A. `coverage-matrix.yaml`** — Responsible for:

- upstream IDs (ACT-#, JTBD-#, JOUR-#, UC-#, EDGE-#, ASM-#, NG-#)
- semantic dimensions and their status
- coverage blockers
- readiness gate

```yaml
# coverage-matrix.yaml
semantic_dimensions:
  problem_clarity:
    status: covered          # covered | partial | missing | not_applicable
    evidence: "Problem framed independently from solution"
  actor_roles_and_permissions:
    status: partial
    gap: "Admin role mentioned but permissions not defined"
  trigger_and_preconditions:
    status: covered
  core_behavior:
    status: covered
  success_signal:
    status: covered
  explicit_non_goals:
    status: missing
    gap: "No NG-# items defined"
  failure_modes:
    status: partial
    gap: "EDGE-1 and EDGE-3 have no downstream resolution"
  degraded_behavior:
    status: missing
  data_mutations:
    status: partial
  side_effects:
    status: missing
  permissions_and_policy:
    status: missing
  observability:
    status: missing
  # Activated by archetype
  archetype_dimensions: {}

mandatory_dimensions:
  - problem_clarity
  - actor_roles_and_permissions
  - core_behavior
  - success_signal
  - explicit_non_goals
  - failure_modes
  - permissions_and_policy

edge_resolution:
  EDGE-1: { status: unresolved, gap: "No US-# or FS-# handles corrupted file" }
  EDGE-2: { status: resolved, via: "US-4 Given/When/Then" }
  EDGE-3: { status: deferred, via: "NG-2 — deferred to Phase 2" }
```

**B. `elicitation-state.yaml`** — Responsible for:

- current module
- current question
- questions already asked
- answer quality per question
- reflection checkpoint status
- user fatigue indicator
- continue/pause decision

```yaml
# elicitation-state.yaml
interview:
  mode: feature
  module: edges
  question_index: 3
  current_question: "What happens if the import is interrupted mid-way?"
  questions_asked: 14
  questions_remaining: 6
  answer_quality_log:
    - { q: "Who uses this?", quality: sufficient, type: confirmed }
    - { q: "What's the main job?", quality: rich, type: confirmed }
    - { q: "How many users?", quality: insufficient, escalated: true }
  reflection_due: true          # triggers at next opportunity
  fatigue_check_at: 15          # offer pause after this many questions
  fatigue_offered: false
  session_paused: false
  active_archetypes: [import_export]
```

**Benefit:** Keeps interview operational state clean and coverage matrix semantically focused.

**Files to create:**
- Schema for `elicitation-state.yaml` in `state_defaults.py`
- Rename/restructure `coverage-matrix.yaml` schema in ideate knowledge

**Priority:** P0
**Effort:** ~120 lines

### R2 — Create a shared Elicitation Engine

Do not duplicate interview logic across all agents.

Create a common mechanism that:

- reads `elicitation-state.yaml`
- knows where the interview stopped
- knows when the answer was insufficient
- applies vague-response escalation
- injects reflection checkpoints
- detects "I don't know", fatigue, and need for deepening

**Minimum rules:**

1. Before asking, read the state
2. If the module is already covered, summarize instead of repeating
3. If the answer was vague, fire a mandatory probe
4. Every 3-4 substantive answers, execute reflection
5. If the user shows fatigue, offer: continue / accept explicit assumptions / pause with persisted checkpoint

**Vague-response escalation table** (shared knowledge file):

```markdown
# .claude/skills/discovery/knowledge/vague-response-escalation.md

| Vague pattern | Mandatory follow-up |
|---|---|
| "must be fast/performant" | "Fast means <200ms? <1s? <5s? What number would make the user complain?" |
| "needs to be secure" | "Secure against which threat? Which data is sensitive? Who must NOT have access?" |
| "easy to use" | "Describe someone who would struggle. What would they try and where would they get stuck?" |
| "needs to scale" | "How many simultaneous users today? In 12 months? In 3 years?" |
| "many/few/several" | "Give me a number. Order of magnitude: tens, hundreds, thousands?" |
| "we'll figure that out later" | "Does this block implementation? If yes, I need a decision now. If no, I'll register as ASM-# open with a deadline." |
| "like [competitor]" | "What exactly from [competitor] do you want? What do you NOT want?" |
| "obvious/standard/normal" | "What's obvious to you may not be to the dev. Explain the 'obvious' as if I'd never seen it." |
```

**"I don't know" classifier:**

When the user says "I don't know", "good question", "never thought about that":

```
"Got it. Help me classify this:
A) Do you know who would know? (I'll register as open question with owner)
B) Is it safe to leave for later? (I'll register as deferred assumption)
C) Could this change the scope of what we're building? (we need to explore before continuing)"
```

**Reflection checkpoint template:**

```
"Before continuing, let me summarize what I understood so far:
1. The problem is [X] because [Y]
2. The main actor is [actor] who currently does [workaround]
3. Success would be measured by [metric]
4. Out of scope: [NG-1, NG-2]
Is this correct? Anything I misinterpreted or that's missing?"
```

**Files to create/change:**
- `.claude/skills/discovery/elicitation-engine/SKILL.md` — new reusable skill (~150 lines)
- `.claude/skills/discovery/knowledge/vague-response-escalation.md` — new knowledge file (~40 lines)
- Each sub-skill (jtbd-extractor, journey-sweep, edge-case-sweep, use-case-matrix, assumption-audit) — add state read/write (~30 lines each)

**Priority:** P0
**Effort:** ~340 lines

### R3 — Discovery must adopt story-based interview as default

Based on external research, this is one of the most important changes.

**Rule:** Whenever possible, discovery must pull: real episode, last concrete case, step by step of what happened, why it mattered, what the person did after.

**Mandatory probes:**

- "Tell me about the last time this happened"
- "Walk me through step by step what you did"
- "What happened after?"
- "Why was that important?"
- "What did you do to work around it?"
- "How much time/money did that workaround cost?"

**Anti-patterns to avoid:**

- "Would you use X?" → always yes. Ask about past behavior instead.
- "Don't you think X is important?" → leading. Ask "What matters most to you?"
- "Is this a problem?" → primes for yes. Ask "Tell me about the last time..."

**Benefit:** Reduces abstraction, rationalization, premature feature requests.

**Files to change:**
- `.claude/agents/discovery-agent.md` — add "Story-Based Interview" section
- `.claude/agents/planning-agent.md` — same section
- `.claude/skills/discovery/project-context/knowledge/guide.md` — rewrite probes to story-based format

**Priority:** P0
**Effort:** ~60 lines

### R4 — Brainstorm must become a structured dynamic in 5 stages

**Problem:** Today brainstorm still risks being too generic.

**Proposal:**

Standardize brainstorm into:

1. **Target Opportunity** — Which specific opportunity or HMW are we solving?
2. **HMW prompts** — Reframe as "How Might We" to open the solution space
3. **Divergence** — Generate as many solutions as possible without judgment
4. **Clustering + selection** — Group by theme, evaluate by feasibility/impact, select 2-3
5. **Assumptions + next tests** — For each selected direction, list assumptions and cheapest validation method

**Initial techniques (v6.0):**

- HMW + clustering
- Crazy 8s
- Gut Check (confidence vote before deep analysis)

**Future expansion:**

- mash-ups
- role play
- reverse brainstorming
- six thinking hats

**Benefit:** Prevents brainstorm from becoming an undisciplined creative conversation.

**Files to change:**
- `.claude/skills/discovery/ideate/SKILL.md` — restructure brainstorm module

**Priority:** P0
**Effort:** ~80 lines

### R5 — Archetype-aware probe packs, starting small

Previous versions proposed 6-12 archetypes. Start with 3 to reduce maintenance risk. Expand after they stabilize.

**Archetype 1 — Workflow / Approval**

| # | Probe | Sufficient answer |
|---|-------|-------------------|
| 1 | What states exist? | Named list with descriptions |
| 2 | Which transitions are valid? | State A → State B with trigger |
| 3 | Which transitions are forbidden? | Explicit "X → Y is not allowed because..." |
| 4 | What SLA/timeout matters? | Number + consequence of breach |
| 5 | Is there delegation or escalation? | Who, when, to whom |
| 6 | Who is notified at each transition? | Channel + recipient per transition |
| 7 | Can a completed flow be reopened? | Yes/No + conditions |
| 8 | Is there an audit trail requirement? | What is logged, who can view |

**Archetype 2 — API / Integration**

| # | Probe | Sufficient answer |
|---|-------|-------------------|
| 1 | How does it authenticate? | Mechanism + scopes |
| 2 | Does it need retry? | Yes/No + strategy |
| 3 | Does it need idempotency? | Key type + dedup window |
| 4 | What rate limit? | Number + behavior when exceeded |
| 5 | What happens on partial failure? | Transaction boundary + rollback |
| 6 | What is the error contract? | Schema + specific codes |
| 7 | Is backward compatibility required? | Versioning strategy |
| 8 | Are there webhooks? | Events + delivery guarantees + replay |

**Archetype 3 — Import / Export**

| # | Probe | Sufficient answer |
|---|-------|-------------------|
| 1 | What formats are accepted? | Explicit list (CSV, XLSX, etc.) |
| 2 | What is the max file size? | Number + behavior when exceeded |
| 3 | Can the user preview before committing? | Yes/No + what's shown |
| 4 | What happens with invalid rows? | Skip/reject/partial + user feedback |
| 5 | Can a completed import be rolled back? | Yes/No + mechanism + time window |
| 6 | Can an interrupted import resume? | Yes/No + mechanism |
| 7 | How are duplicates handled? | Skip/overwrite/create new/ask user |
| 8 | Is there a progress indicator? | Type + cancellation support |
| 9 | What's the expected volume? | Rows per import + frequency |
| 10 | Who can import? | Permission rules |

**Integration:** After the "Problem Framing" module in ideation, ask: "Which of these patterns most resembles what we're building?" Load 1-3 probe packs. Integrate probes into existing modules.

**Future expansion (after v6.0 stabilizes):** billing/subscription, AI/copilot, CRUD/admin, notifications, search/filter.

**Files to create:**
- `.claude/skills/discovery/elicitation-archetypes/workflow-approval.md`
- `.claude/skills/discovery/elicitation-archetypes/api-integration.md`
- `.claude/skills/discovery/elicitation-archetypes/import-export.md`
- `.claude/skills/discovery/ideate/SKILL.md` — add archetype selection step

**Priority:** P0
**Effort:** ~400 lines

### R6 — Mandatory semantic coverage by dimension

Replace the "quantity" gate with a gate that evaluates whether we covered:

- problem clarity
- trigger and preconditions
- core behavior
- success signal
- anti-requirements
- failure modes
- degraded behavior
- actors and permissions
- data mutations
- side effects
- observability

And when archetype is active: archetype-specific dimensions.

**Rule:** No mandatory dimension can remain `false` without: `not_applicable` justification OR explicit registration as assumption/open question.

**EDGE-# resolution integration:** The `failure_modes` dimension is only `covered` when ALL identified EDGE-# items have either:

- a resolution (US-# Given/When/Then, FS-# rule, ADR, API error code), OR
- an explicit deferral as NG-# with rationale

This prevents the scenario where `failure_modes: covered` because some edge cases were handled while others were silently dropped.

**Gate behavior change:**
- Current: "edge_cases >= 3 → pass"
- New: "failure_modes dimension == covered AND all EDGE-# resolved or deferred → pass"

**Files to change:**
- `coverage-matrix.yaml` — add `semantic_dimensions` + `edge_resolution`
- `ideate/SKILL.md` — drive interview by uncovered dimensions
- `validate-gate/SKILL.md` — fail on missing mandatory dimensions + unresolved EDGE-#
- `check-traceability/SKILL.md` — add EDGE-# resolution chain
- `state_defaults.py` — add dimensions to default

**Priority:** P0
**Effort:** ~200 lines

### R8 — `/elicit` command as semantic stress-test

**When to use:**

1. After `ideate`, before `feature` or `discover`
2. After `requirements`, before `design`
3. When the user says: "I want to challenge what's been specified"

**What it does:** Does not generate new product artifacts. It interrogates existing ones.

**Interrogation techniques:**

1. **Goal vs Non-Goal clash** — For each NG-#, verify whether any downstream doc contradicts it
2. **Actor/Permission mismatch** — For each actor, verify whether permitted actions are consistent across docs
3. **Missing lifecycle states** — For each entity with states, verify all transitions (including forbidden)
4. **EDGE-# resolution audit** — For each EDGE-#, verify resolution exists downstream
5. **Developer simulation** — "If I were implementing [feature], what would I still need to know?"
6. **Vague quantifier check** — Scan for "fast", "secure", "easy", "many" without metrics
7. **Cross-doc consistency** — DB constraints vs data-dictionary vs API schema vs feature-spec
8. **Example challenge** — "Give me one canonical success example, one ugly edge example, and one 'this should never happen' example"

**Output:** A findings report (EL-# items), not a new source of truth.

**False positive awareness:** The validator must differentiate: real placeholder vs template example vs quoted text vs reference docs.

**Files to create:**
- `.claude/commands/elicit.md` (~60 lines)
- `.claude/skills/validation/elicit/SKILL.md` (~150 lines)

**Priority:** P0
**Effort:** ~210 lines

### R9 — Developer Handoff Packet with explicit derivation rule

**Design rule:** The packet must be: generated automatically, derived from authoritative sources, regenerable, non-authoritative.

**Minimum sections:**

1. Context and objective (from feature-brief + product-vision)
2. Non-goals NG-# (from feature-brief + NG-# registry)
3. Actors and permission matrix (from feature-spec or tech-spec)
4. Business rules (from feature-brief + feature-spec)
5. Acceptance criteria with Given/When/Then (from user-stories)
6. Error handling matrix ES-# (from feature-spec)
7. State machine if applicable (from feature-spec)
8. API contracts relevant to feature (from api-spec)
9. Data mutations and validation rules (from database-spec + data-dictionary)
10. Performance requirements NFR-# (from tech-spec)
11. Observability requirements (from quality-checklist)
12. Implementation tasks impl-# (from implementation-plan)
13. Unresolved blockers (from feature-brief + ASM-# open)

**Upstream dependency rule:** The packet pulls from upstream docs. It does not generate content that doesn't exist upstream. This means upstream templates must be expanded FIRST (see R10).

If an upstream section is missing, the packet flags it as `⚠️ MISSING — [source doc] does not contain this section. Run /elicit to identify the gap.`

**Scope:** One packet per feature (or per sprint scope), not per project.

**Files to create:**
- `.claude/skills/implementation/implementation-packet/SKILL.md` (~200 lines)
- `.claude/skills/implementation/implementation-packet/knowledge/template.md` (~80 lines)

**Priority:** P0
**Effort:** ~280 lines

### R11 — Executable anti-placeholder and anti-vagueness validation

Maintained, with an important improvement:

**False positive care:** Simple regex is not enough. The validator must differentiate: real placeholder, template example, quoted text, reference docs.

**Priority checks:**

1. Gherkin placeholder — `[USER]`, `[ACTION]`, `[RESULT]` or clauses with <5 words
2. NFR without number + unit — "fast", "secure" without metric
3. Vague vocabulary without metric — flag and suggest replacement
4. Mandatory section empty — auto-fail
5. Missing minimum error scenario — for each happy-path US-#, require ≥1 error US-# or EDGE-# resolution

**Vague terms to flag:**

| Term | Replacement guidance |
|------|---------------------|
| "fast" | "response time < X ms at p95" |
| "easy to use" | "new user completes [task] without help in < X min" |
| "secure" | specific threat model + encryption + auth |
| "scalable" | "supports X concurrent users / Y req/s" |
| "real-time" | "latency < X ms" |
| "robust" | "handles [specific failures] with [specific behavior]" |
| "seamless" | specific integration/transition requirements |

**Files to change:**
- `validate-gate/SKILL.md` — add anti-placeholder + anti-vagueness checks to Layer 1 and Layer 2

**Effort:** ~60 lines

### R13 — "I don't know" differentiation

When the user says "I don't know", ETUS must classify before proceeding:

| Type | Meaning | Treatment |
|------|---------|-----------|
| "I don't know and need to research" | Aware of gap | ASM-# open with owner + deadline |
| "I don't know and it's OK not to know now" | Conscious deferral | ASM-# deferred with justification |
| "I never thought about that" | Didn't know the question existed | **Most valuable — deepen before moving on** |

Already covered in R2 (elicitation engine). Listed here to emphasize it deserves formal treatment, not just a prompt hint.

**Priority:** P1
**Effort:** Included in R2

---

### R14 — User fatigue management

After a certain number of substantive questions, ETUS must offer:

- continue deepening
- pause and resume later (checkpoint persisted in elicitation-state.yaml)
- accept explicit assumptions for remaining dimensions

These assumptions must be clearly marked as: `assumed_default` + `revisit_required`.

**Rule:** Blocking dimensions (problem_clarity, failure_modes, permissions_and_policy) cannot be accepted as defaults. Non-blocking dimensions (observability, side_effects) can.

Already integrated into R2 (elicitation engine). Listed here because the blocking/non-blocking distinction is important.

**Priority:** P1
**Effort:** Included in R2

### R15 — Correct-course cascade with automatic propagation

**Problem:** After change-proposal approval, downstream documents are not updated. The user must manually fix 5-10 files.

**Mechanism:**

1. User approves change-proposal.md
2. `/correct-course` **greps** all project docs for IDs mentioned in the change (removed PRD-F-#, changed US-#, etc.)
3. **Lists** every document that references an affected ID, with the specific line
4. **Generates** proposed diffs for each affected document (remove reference, update wording, change priority)
5. **Presents** before/after for user approval (one doc at a time)
6. **Applies** approved changes
7. **Updates** ids.yml, coverage-matrix, workflow-state, feature-status
8. If execution adapter enabled, **flags** which external issues need updates

**Files to change:**
- `.claude/skills/planning/correct-course/SKILL.md` — add reconciliation phase (~50 lines)
- `.claude/commands/correct-course.md` — expand command to include cascade (~15 lines)

**Priority:** P1
**Effort:** ~65 lines

### R16 — Post-implementation feedback loop

**Goal:** Discover which questions were missing, which assumptions the dev had to invent, and in which phase that should have been captured.

**Mechanism:**

Create `/retro` command:
1. Ask the dev: "What questions did you have to answer on your own during implementation?"
2. For each answer, classify: "This should have been asked in which phase?"
3. Save as learning in `docs/ets/projects/{project-slug}/learnings/`
4. If pattern repeats 2+ times, suggest adding to the corresponding archetype probe pack

**Value:** This is what transforms archetype packs into assets that improve over time.

**Files to create:**
- `.claude/commands/retro.md` (~30 lines)
- `.claude/skills/retrospective/dev-feedback/SKILL.md` (~70 lines)

**Priority:** P1-P2
**Effort:** ~100 lines

## Success Metrics

### Primary metrics

1. **Questions the dev still had to ask after receiving the packet**
   Target: `< 3` per feature

2. **Semantic coverage before gate**
   Target: `> 80%` on first pass

3. **Archetype probes answered with sufficient response**
   Target: `> 70%`

4. **Vague answers escalated correctly**
   Target: `> 90%` of vague responses generate a follow-up probe

5. **Iterations per gate**
   Target: average `< 1.5`

6. **Rejected changes reappearing downstream**
   Target: `0`

7. **EDGE-# resolution rate before Implementation Readiness Gate**
   Target: `100%` resolved or explicitly deferred

8. **Feedback loop yield**
   Target: `> 3` new archetype probes added per quarter from `/retro` learnings

### Gold metric

> How many product/behavior decisions did the dev have to invent because the documentation didn't answer?

This is probably the truest measure of the system.

## Definition of Done

A feature or product is only interview-complete when:

1. The problem is clear independently from the proposed solution
2. Archetype-specific probes have been exhausted for all applicable archetypes
3. All mandatory semantic dimensions are covered or justified as not-applicable
4. Actors, operators, and approvers are explicit with permission matrix
5. Non-goals are explicit and traceable as NG-# IDs
6. Every EDGE-# has a resolution (US-#, FS-#, ADR, API error code) or explicit deferral as NG-#
7. Active assumptions are classified (open/deferred/assumed-default) with owners and deadlines
8. The spec has been stress-tested via `/elicit` with zero unresolved EL-# blockers
9. Error handling matrix (ES-#), state machines, validation rules, and observability are documented in upstream sources
10. Everything is consolidated into a single implementation packet (derived, not authoritative)
11. Remaining unknowns are explicit enough that engineering knows what to answer before coding
12. Gate decisions and rejected approaches are persisted for downstream phases

## Final Conclusion

ETUS does not need to become a "heavier" framework.

It needs to become a framework that is more:

- explicit in interview state
- semantic in coverage
- disciplined in brainstorm
- mechanical in validation
- safe in developer handoff
- resilient to changes
- honest about what it doesn't know yet

If v1 showed the problem, v2 showed the direction, and v3 consolidated the path, v4 delivers the complete spec:

> **The future of ETUS is to move from a good documented methodology to an operational system of elicitation, decision, and handoff.**
