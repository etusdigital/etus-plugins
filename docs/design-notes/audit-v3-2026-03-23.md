# Audit v3: ETUS PMDocs — From Interview to Developer Handoff

**Date:** 2026-03-23
**Framework version:** v5.3
**Base:** Full read of 9 agents, 45 skills, 13 commands, 5 gates, 6 hooks, state_defaults.py, coverage-matrix.yaml
**Method:** Cross-reference of three independent audits (strategic, tactical, file-level verification) + original ideas from v2

---

## Core Diagnosis

ETUS PMDocs has the right foundation: ideation before specification, coverage matrix, traceability chain, SST validation, 4 modes, gates between phases. This is better than 90% of product documentation frameworks.

The problem is not missing methodology. The problem is that **output quality depends on prompt discipline, not mechanical enforcement**.

In practice, this means three things:

1. **The interview can finish "complete" without asking the right questions for the type of problem.** The framework requires ">= 2 edge cases" — but it does not distinguish between generic edge cases and domain-critical ones. An upload feature with 2 edge cases about "duplicate name" and "empty field" passes validation, but misses "corrupted file", "partial upload", "size limit", "virus scan".

2. **Validation checks structure, not semantics.** A user story with `Given [USER], When [ACTION], Then [RESULT]` passes the traceability check because it has Given/When/Then structure. But it is placeholder.

3. **The developer receives 10+ documents and must assemble the puzzle alone.** There is no consolidated artifact that says: "here is everything you need to implement this feature, in one place."

---

## What Already Works Well (preserve these)

Before recommendations, it is important to fix what must NOT change:

**1. Coverage-first ideation** — The ideate skill already forces: problem framing before solutions, actors before specs, journeys before architecture. The sequence `ingest → problem → actors → jobs → journeys → cases → edges → assumptions → brainstorm → synth` is correct.

**2. Anti-requirements exist** — The feature-brief (Q4: "What is out of scope?") and the opportunity-pack ("constraints and guardrails" section) already capture anti-requirements. They are not first-class traceable objects yet, but they exist.

**3. Traceability chain** — `BO-# → PRD-F-# → US-# → FS-# → impl-#` with automated validation via check-traceability. This is a real competitive advantage.

**4. Pressure test in feature-brief** — The skill already has a "PRESSURE TEST" section with 4 challenge questions. It is not enforcement, but it is a quality instruction.

**5. Coverage matrix with traceable IDs** — `ACT-#`, `JTBD-#`, `JOUR-#`, `UC-#`, `EDGE-#`, `ASM-#` with status tracking and downstream linkage. The tracking infrastructure exists.

**6. Contradiction register in ideation** — "If sources disagree, record the contradiction instead of silently resolving it." The intent already exists; enforcement is missing.

## Recommendations (ordered by impact × effort)

### R1. Elicitation Engine with persistent state

**Concrete problem:** The ideate skill has 11 modules, but progression between them depends on the agent "remembering" where it stopped. If context is lost (long session, session switch), the agent may skip modules or repeat questions.

**Evidence in code:** `coverage-matrix.yaml` has `step_status` per module (not_started/in_progress/covered), but no skill checks this state before acting. The skill says "checkpoint after each semantic block" but the checkpoint mechanic is: save the YAML and hope the next agent reads it.

**What is concretely missing:**

Each module in coverage-matrix needs additional fields:

```yaml
step_status:
  actors:
    status: in_progress
    questions_asked: 3
    questions_remaining: 2
    current_question: "Who blocks or approves this workflow?"
    answers_quality: sufficient  # insufficient | sufficient | rich
    last_answer_timestamp: "2026-03-23T14:30:00Z"
```

And each skill needs an entry rule:

```
BEFORE asking any question:
1. Read coverage-matrix.yaml
2. Find the corresponding module
3. If status == "covered" → skip, summarize what already exists
4. If status == "in_progress" → continue from current_question
5. If status == "not_started" → start from the beginning
```

**Files to change:**
- `.claude/skills/discovery/ideate/knowledge/coverage-matrix.yaml` — add per-step state fields
- `.claude/skills/discovery/ideate/SKILL.md` — add mandatory entry rule
- Each sub-skill (jtbd-extractor, journey-sweep, edge-case-sweep, use-case-matrix, assumption-audit) — add state read

**Effort:** ~80 lines YAML + ~30 lines per sub-skill = ~230 lines total

### R2. Archetype-aware probe packs

**Concrete problem:** The ideate skill asks "invalid states?", "dependency failure?", "race condition?" for every feature type. These are generic categories. A workflow feature needs questions about states/transitions/SLA/escalation. An import/export feature needs questions about formats/size/rollback/resumability.

**Evidence in code:** `edge-case-sweep/SKILL.md` has 8 fixed categories: invalid input, dependency unavailable, race condition, permission mismatch, partial success, abuse, rollback, observability. Good generic categories, but they don't cover the specific gaps for each feature type.

**What to do:**

Create a step zero in ideation: classify the work item into 1-3 archetypes, then load specific probe packs.

Proposed archetypes (start with 8, expand later):

| Archetype | Mandatory probes |
|-----------|-----------------|
| **CRUD/backoffice** | Roles, permissions, audit trail, bulk operations, soft delete, search/filter, pagination, export |
| **Workflow/approval** | States, transitions, forbidden transitions, timeout/SLA, delegation, escalation, reopen, audit trail, notifications per transition |
| **Import/export** | Accepted formats, max size, malformed rows, preview, validation, rollback, resumability, encoding, progress indicator, duplicate handling |
| **API/integration** | Auth, retry, rate limit, idempotency, partial failure, webhook replay, versioning, backward compat, error contract |
| **AI/copilot** | Input boundaries, hallucination risk, human override, refusal rules, evaluation criteria, fallback, acceptable latency, cost per invocation |
| **Billing/subscription** | Lifecycle, upgrade/downgrade, proration, grace period, dunning, refund, invoice, tax compliance |
| **Notifications/messaging** | Channels (email/push/in-app/SMS), frequency caps, opt-out, delivery guarantees, templates, localization |
| **Search/filter/discovery** | Indexing strategy, ranking, facets, autocomplete, fuzzy matching, zero-results handling, personalization |

**Format for each probe pack:** A markdown file in `.claude/skills/discovery/elicitation-archetypes/{archetype}.md` with:
- 8-12 mandatory questions (one per message)
- For each question: what is "sufficient answer" vs "needs deepening"
- Archetype-specific anti-patterns

**Integration:**
1. After the "Problem Framing" module in ideation, ask: "Which of these patterns most resembles what we're building? [archetype list]"
2. Load 1-3 probe packs
3. Integrate questions into existing modules (edge-case-sweep inherits archetype probes)

**Example probe pack — Import/Export:**

```markdown
## Archetype: Import/Export — Mandatory Probes

1. "What file formats must be supported? (CSV, XLSX, JSON, XML, other?)"
2. "What is the maximum file size? What happens if exceeded?"
3. "When a row has invalid data: skip the row? reject the entire file? import valid rows only?"
4. "Can the user preview data before committing the import?"
5. "What validation rules apply per field? (format, uniqueness, referential integrity)"
6. "What happens if the import is interrupted? (network failure, browser close, timeout)"
7. "Can the user undo/rollback a completed import?"
8. "How are duplicate records handled? (skip, overwrite, create new, ask user)"
9. "Is there a progress indicator? Can the user cancel mid-import?"
10. "What is the expected volume? (10 rows? 10K? 1M?) How does that affect UX and architecture?"
11. "Who can import? Are there permission restrictions?"
12. "Is there an audit trail of who imported what and when?"
```

**Effort:** ~120 lines per archetype × 8 = ~960 lines + ~40 lines in ideate skill

### R3. Semantic coverage dimensions instead of counting

**Concrete problem:** The current threshold is `>= 3 edge cases` for Product mode. But 3 edge cases about "empty field" are worth less than 1 edge case about "what happens when the payment service is down".

**Evidence in code:** `coverage-matrix.yaml` has `count: 0` and `threshold: 0` per domain. The ideation gate checks: `edge_cases_covered: false/true`. This is binary and count-based.

**What to do:**

Replace count thresholds with **mandatory semantic dimensions**. For each work item, track whether these dimensions are covered (yes/no/not-applicable):

```yaml
semantic_dimensions:
  problem_clarity: false           # Is the problem described independently from the solution?
  actor_roles_and_permissions: false   # Who can do what?
  trigger_and_preconditions: false     # What starts the flow and what must be true before?
  core_behavior: false             # What does the system do in the happy path?
  success_signal: false            # How does the user know it worked?
  explicit_non_goals: false        # What must NOT happen?
  failure_modes: false             # What happens when things go wrong?
  degraded_behavior: false         # How does the system work at reduced capacity?
  data_mutations: false            # What data is created/changed/deleted?
  side_effects: false              # Notifications, events, webhooks, logs?
  permissions_and_policy: false    # Access rules, compliance, rate limits?
  observability: false             # Logs, metrics, alerts needed?

  # Optional dimensions (activated by archetype)
  archetype_dimensions: {}
```

**Gate rule:** The ideation readiness gate fails if any mandatory dimension is `false` without a `not_applicable` justification.

**Archetype integration:** Each archetype adds extra dimensions to `archetype_dimensions`. Example for workflow: `state_machine: false`, `invalid_transitions: false`, `sla_timeout: false`.

**Files to change:**
- `coverage-matrix.yaml` — add `semantic_dimensions`
- `ideate/SKILL.md` — swap count thresholds for dimensions
- `validate-gate/SKILL.md` — add dimension check
- `state_defaults.py` — add dimensions to default

**Effort:** ~60 lines YAML + ~40 lines per affected skill = ~200 lines total

### R4. Non-goals as traceable objects (NG-#)

**Concrete problem:** The feature-brief asks "What is out of scope?" (Q4), the opportunity-pack has "constraints and guardrails", but neither is traceable downstream. If a user story or impl-plan silently reintroduces something declared out of scope, nobody detects it.

**Evidence in code:** `check-traceability/SKILL.md` traces `BO-# → PRD-F-# → US-# → FS-# → impl-#` but does NOT trace non-goals. The `ID_PATTERNS` in `state_defaults.py` does not include `NG-#`.

**What to do:**

1. Add `NG-#` to `ID_PATTERNS`:
```python
"non_goals": r"\bNG-\d+\b",
```

2. Each `NG-#` must have:
- statement: what must NOT happen
- reason: why it was excluded
- adjacent_behavior: which valid functionality borders this non-goal
- scope: `permanent | deferred_to_v2 | conditional`
- downstream_must_respect: list of docs that cannot contradict

3. Add check in `check-traceability`:
- If a downstream doc mentions behavior that contradicts an `NG-#`, flag as violation

4. Add "Non-Goals" section to opportunity-pack template and feature-brief template

**Files to change:**
- `state_defaults.py` — add NG-# to ID_PATTERNS
- `feature_lifecycle.py` — extract NG-# from docs
- `check-traceability/SKILL.md` — add non-goal violation check
- `ideate/knowledge/template.md` — add NG-# section
- `planning/feature-brief/knowledge/template.md` — add NG-# section

**Effort:** ~120 lines total

### R5. `/elicit` command — Stress-test before locking the spec

**Concrete problem:** There is no dedicated phase to "interrogate the spec until ambiguity is gone". The closest pieces are scattered: pressure test in feature-brief, adversarial review in validate-gate, edge-case-sweep in ideation. None is focused on "finding what remained implicit".

**What to do:**

Create `/elicit` as a standalone command, usable:
- After `/ideate`, before `/feature`
- After `/plan requirements`, before `/design`
- At any point when the user wants to challenge what was documented

`/elicit` does NOT create new artifacts. It interrogates existing ones and generates a findings list:

```markdown
# Elicitation Findings

## Ambiguities Found
- EL-1: US-3 says "notify the user" but does not specify channel (email? push? in-app?)
- EL-2: feature-brief says "admin can edit" but does not define whether edit includes delete

## Contradictions Found
- EL-3: PRD says "CSV only" (PRD-F-2), api-spec accepts XLSX (endpoint POST /import)
- EL-4: user-story US-5 says "synchronous confirmation", feature-spec says "eventual consistency"

## Missing Boundaries
- EL-5: No document defines what happens when the rate limit is reached
- EL-6: "editor" vs "viewer" permissions never formalized in a matrix

## Hidden Assumptions
- EL-7: All docs assume single-tenant but none makes this explicit
- EL-8: API spec assumes UTC but frontend may be in BRT

## Questions a Developer Would Still Ask
- EL-9: "If I receive US-3 to implement, what would I still need to ask?"
```

**Interrogation techniques:**

1. **Goal vs Non-Goal clash** — For each NG-#, verify whether any downstream doc contradicts it
2. **Actor/Permission mismatch** — For each actor, verify whether permitted actions are consistent across docs
3. **Missing lifecycle states** — For each entity with states, verify whether all transitions are covered (including forbidden ones)
4. **Missing failure handling** — For each UC-#, verify whether at least 1 EDGE-# exists
5. **Developer simulation** — "If I were implementing [feature], what would I still need to know?"
6. **Vague quantifier check** — Scan for words like "fast", "secure", "easy", "many", "few" without metrics
7. **Cross-doc consistency** — DB constraints vs data-dictionary vs API schema

**Files to create:**
- `.claude/commands/elicit.md` — ~60 lines
- `.claude/skills/validation/elicit/SKILL.md` — ~150 lines

**Effort:** ~210 lines

### R6. Implementation Packet — consolidated document for the developer

**Concrete problem:** The framework generates opportunity-pack, feature-brief, user-stories, design-delta, tech-spec, api-spec, data-dictionary, implementation-plan. The developer must open 8 files and mentally synthesize what to implement.

**Evidence in code:** The implementation-plan (`SKILL.md`) generates `impl-#` tasks with dependencies, but does NOT consolidate business rules, error handling, permissions, data validation in one place.

**What to do:**

Create an `implementation-packet` skill that generates a single document per feature (or per sprint):

```markdown
# Implementation Packet: [Feature Name]

## 1. Context (2 paragraphs)
Extracted from: feature-brief problem statement + product-vision BO-#

## 2. Non-Goals (NG-#)
Extracted from: feature-brief out-of-scope + NG-# registry

## 3. Actors and Permission Matrix
| Actor | Can do | Cannot do |
Extracted from: opportunity-pack actor map + tech-spec NFRs

## 4. Business Rules (FB-#, business rules)
Extracted from: feature-brief + feature-spec

## 5. Acceptance Criteria (US-# with Given/When/Then)
Extracted from: user-stories.md

## 6. Error Handling Matrix (ES-#)
| Scenario | Trigger | System Response | User Message | Retry? | Rollback? |
Extracted from: EDGE-# + user-stories error scenarios + tech-spec NFRs

## 7. State Machine (if applicable)
Mermaid diagram + valid transitions + forbidden transitions + side effects per transition
Extracted from: feature-spec

## 8. API Contracts
Relevant endpoints with request/response schemas + error responses + idempotency + timeout
Extracted from: api-spec.md (filtered by feature)

## 9. Data Mutations and Validation Rules
Which tables/fields are created/changed/deleted + validation rules per field (format, regex, min/max, required, examples)
Extracted from: database-spec + data-dictionary (filtered by feature)

## 10. Performance Requirements
Applicable NFR-# targets for this feature
Extracted from: tech-spec.md

## 11. Observability Requirements
Mandatory logs (event, level, fields), metrics (name, type, labels), alerts (condition, severity, channel, runbook)
Generated by: this skill (new content — does not exist in upstream docs today)

## 12. Tasks (impl-#)
Implementation sequence with dependencies
Extracted from: implementation-plan.md

## 13. Unresolved Questions
Open questions the dev must resolve BEFORE implementing
Extracted from: feature-brief outstanding questions + ASM-# open
```

**Critical rule:** This document is GENERATED automatically, not written manually. It pulls from each upstream doc and consolidates. If an upstream changes, the packet can be regenerated.

**Upstream dependency note:** Sections 3, 6, 7, 9, and 11 depend on content that upstream skills do not yet generate in structured form:

| Packet section | Upstream skill | What to add to upstream template |
|----------------|---------------|----------------------------------|
| 3. Permission Matrix | tech-spec or feature-spec | Mandatory permission matrix section |
| 6. Error Handling Matrix (ES-#) | feature-spec | Mandatory ES-# section (see R14) |
| 7. State Machine | feature-spec | Mandatory state diagram for features with >2 states |
| 9. Validation Rules | data-dictionary | Mandatory format/regex/min/max/examples per field |
| 11. Observability | quality-checklist | Mandatory logs/metrics/alerts section |

These upstream template expansions should be implemented alongside or before the packet skill.

**Files to create:**
- `.claude/skills/implementation/implementation-packet/SKILL.md` — ~200 lines
- `.claude/skills/implementation/implementation-packet/knowledge/template.md` — ~80 lines

**Files to modify (upstream templates):**
- `.claude/skills/planning/feature-spec/knowledge/template.md` — add ES-#, state machine, permission matrix
- `.claude/skills/data-design/data-dictionary/SKILL.md` — add validation rules per field
- `.claude/skills/implementation/quality-checklist/SKILL.md` — add observability section

**Effort:** ~280 lines (packet) + ~120 lines (upstream templates) = ~400 lines total

### R7. Anti-placeholder and anti-vagueness validation in gates

**Concrete problem:** A user story with `Given [USER], When [ACTION], Then [RESULT]` passes validation because it has correct structure. An NFR with "API should be fast" passes because it has `NFR-#` format.

**Evidence in code:** `validate-gate/SKILL.md` Layer 1 (Structure) checks "No unresolved [TODO] or [TBD] placeholders". But `[USER]`, `[ACTION]`, `[RESULT]` are not detected as placeholders. Layer 2 (Content) says "LLM-based review" but without specific rejection criteria.

**What to add:**

1. **Anti-placeholder patterns** — Automatically reject:
   - Text in brackets: `[SOMETHING]`, `[TBD]`, `[TODO]`, `[PLACEHOLDER]`
   - Vague words in NFRs without numbers: "fast", "quick", "secure", "easy", "scalable", "reliable"
   - Given/When/Then clauses with fewer than 5 words (indicates unfilled template)

2. **Quantifier check for NFRs** — Every NFR-# must have at least one number (latency in ms, uptime in %, throughput in req/s). If not, flag as `NFR_VAGUE`.

3. **Error path coverage** — For each happy-path US-#, verify that at least 1 US-# or EDGE-# exists with a corresponding error scenario.

**Vague terms to flag:**

| Term | Replace with |
|------|-------------|
| "fast" | "response time < X ms at p95" |
| "easy to use" | "new user completes [task] without help in < X minutes" |
| "secure" | specific threat model + encryption + auth requirements |
| "scalable" | "supports X concurrent users / Y requests per second" |
| "real-time" | "latency < X ms" |
| "robust" | "handles [specific failure modes] with [specific behavior]" |
| "seamless" | specific integration/transition requirements |

**Where to add:**
- `validate-gate/SKILL.md` Layer 1 — add anti-placeholder checks
- `validate-gate/SKILL.md` Layer 2 — add vagueness and error coverage checks

**Effort:** ~60 lines

### R8. Reflection checkpoints ("What I understood is...")

**Concrete problem:** Agents ask, the user answers, the agent moves on. There is no moment of "let me verify I understood correctly". This causes two problems: the agent may misinterpret, and the user may realize their own answer was incomplete.

**Evidence in code:** No agent or skill has a reflection checkpoint instruction. The interaction protocol says "section-by-section approval" for output, but not for input (user answers).

**What to add:**

Simple rule in each agent: **Every 3-4 user answers, pause and summarize:**

```
"Before continuing, let me summarize what I understood so far:

1. The problem is [X] because [Y]
2. The main affected actor is [actor] who currently does [workaround]
3. Success would be measured by [metric]
4. Out of scope: [NG-1, NG-2]

Is this correct? Anything I misinterpreted or that's missing?"
```

**Where to add:**
- `.claude/agents/discovery-agent.md` — add "Reflection Checkpoints" section
- `.claude/agents/planning-agent.md` — same section
- `.claude/skills/discovery/ideate/SKILL.md` — add to INTERACTION PROTOCOL
- `.claude/skills/planning/feature-brief/SKILL.md` — add between Q3 and Q4

**Effort:** ~40 lines total (10 per file)

### R9. Escalation table for vague responses

**Concrete problem:** When the user says "it should be fast", the agent accepts. It should escalate with a specific follow-up question.

**Evidence in code:** The jtbd-extractor has "Do not accept 'save time' without specifying where and how" — but that is an instruction, not a mechanism. There is no pattern table of vague response → follow-up.

**What to add:**

Reusable table in a shared knowledge file:

```markdown
# .claude/skills/discovery/knowledge/vague-response-escalation.md

| Vague pattern | Mandatory follow-up |
|---|---|
| "must be fast/performant" | "Fast means <200ms? <1s? <5s? What number would make the user complain?" |
| "needs to be secure" | "Secure against which threat? Which data is sensitive? Who must NOT have access?" |
| "easy to use" | "Describe someone who would struggle. What would they try and where would they get stuck?" |
| "needs to scale" | "How many simultaneous users today? In 12 months? In 3 years?" |
| "many/few/several" | "Give me a number. Order of magnitude: tens, hundreds, thousands?" |
| "we'll figure that out later" | "Does this block implementation? If yes, I need a decision now. If no, I'll register it as ASM-# open." |
| "like [competitor]" | "What exactly from [competitor] do you want? What do you NOT want?" |
| "obvious/standard/normal" | "What's obvious to you may not be to the dev. Explain the 'obvious' as if I've never seen it." |
```

Referenced by each discovery sub-skill.

**Effort:** ~40 lines (file) + ~5 lines per sub-skill (reference) = ~70 lines

### R11. Post-implementation feedback loop

### R12. Differentiating "I don't know"

**Concrete problem:** When the user answers "I don't know", the framework treats it as a uniform response. But there are three types of "I don't know" with different treatments:

| Type | Meaning | Correct treatment |
|---|---|---|
| "I don't know and need to research" | The user is aware of the gap | Register as ASM-# open with owner + deadline |
| "I don't know and it's OK not to know now" | Conscious decision to defer | Register as ASM-# deferred with justification |
| "I never thought about that" | The user didn't know the question existed | This is the most valuable case — deepen before moving on |

**What to add:**

Instruction in agents: when the user says "I don't know", "good question", "never thought about that":

```
"Got it. Help me classify this:
A) Do you know who would know? (I'll register as an open question with an owner)
B) Is it safe to leave for later? (I'll register as a deferred assumption)
C) Could this change the scope of what we're building? (we need to explore more before continuing)"
```

**Where to add:**
- `.claude/agents/discovery-agent.md` — "Handling Uncertainty" section
- `.claude/agents/planning-agent.md` — same section
- `.claude/skills/discovery/assumption-audit/SKILL.md` — expand AUDIT QUESTIONS

**Effort:** ~30 lines

### R13. Adaptation to user fatigue

**Where to add:**

### R14. EDGE-# resolution tracking

**Concrete problem:** An edge case like "EDGE-1: File exceeds 100MB" is identified during ideation. But no user story, feature-spec rule, or API error code addresses it. The edge case was discovered but never resolved. The current `check-traceability` validates `BO-# → PRD-F-# → US-#` but does not validate `EDGE-# → resolution`.

**Evidence in code:** `check-traceability/SKILL.md` has chains for `ACT/JTBD/JOUR/UC/EDGE → downstream`. The EDGE chain checks if EDGE IDs appear downstream, but does not verify whether they have a **resolution** (a Given/When/Then, a FS-# rule, an ADR, or an API error code that handles the edge case).

**What to add:**

Extend check-traceability with an EDGE resolution rule:

```
For each EDGE-#:
  MUST have resolution in at least one of:
    - user-stories.md (Given/When/Then that handles the edge case)
    - feature-spec-*.md (FS-# rule for handling)
    - tech-spec.md (ADR or NFR for handling)
    - api-spec.md (error response code)

  OR explicitly deferred:
    - Marked "Deferred to [phase]" with rationale and NG-# reference
    - Tracked in non-goals or backlog

  If unresolved and not deferred:
    - Flag as CRITICAL gap in validation report
    - Block Requirements Gate and Implementation Readiness Gate
```

**Where to change:**
- `check-traceability/SKILL.md` — add EDGE-# resolution chain (~20 lines)
- `validate-gate/SKILL.md` — block on unresolved EDGE-# at Requirements and Implementation gates (~10 lines)

**Effort:** ~30 lines

### R15. Correct-course cascade

**Concrete problem:** If a requirement changes mid-project (PRD-F-3 is descoped, a business rule changes, a non-goal is reversed), the `/correct-course` command generates a change-proposal.md but does not guarantee downstream documents are updated. User-stories, feature-specs, design-delta, impl-plan, release-plan, and quality-checklist may still reference the old requirement.

**Evidence in code:** `correct-course/SKILL.md` reads all ENRICHES documents to assess impact and generates a change-proposal.md. But it does not modify any downstream doc. The user must manually update 5-10 documents.

**What to do:**

After change-proposal approval, add a reconciliation phase:

1. **Grep** all project docs for IDs mentioned in the change-proposal (removed PRD-F-#, changed US-#, etc.)
2. **List** every document that references an affected ID, with the specific line
3. **Generate** proposed diffs for each affected document (remove reference, update wording, change priority)
4. **Present** before/after for user approval (one doc at a time)
5. **Apply** approved changes
6. **Update** ids.yml, coverage-matrix, workflow-state, and feature-status
7. If execution adapter is enabled, **flag** which external issues need updates

**Where to change:**
- `.claude/skills/planning/correct-course/SKILL.md` — add reconciliation phase after approval (~50 lines)
- `.claude/commands/correct-course.md` — expand to include cascade logic (~15 lines)

**Effort:** ~65 lines

### R16. Upstream template expansion for developer-critical content

**Concrete problem:** The Implementation Packet (R6) needs to pull content from upstream docs. But several sections depend on content that upstream skills do not yet generate in structured form. Without expanding upstream templates, those packet sections will be empty or LLM-hallucinated.

**What to add to each upstream template:**

| Upstream skill | New mandatory section | Content |
|---------------|----------------------|---------|
| **feature-spec** | Error Handling Matrix (ES-#) | `Scenario \| Trigger \| System Response \| User Message \| Retry? \| Rollback?` per error scenario |
| **feature-spec** | State Machine (if >2 states) | Mermaid diagram + valid transitions + forbidden transitions + side effects per transition |
| **feature-spec** | Permission Matrix | `Action \| Role A \| Role B \| Role C` with ✅/❌ per cell |
| **data-dictionary** | Validation Rules per field | Required/Optional, Format (regex or enum), Min/Max length, Default, Valid example, Invalid example, Behavior on invalid |
| **quality-checklist** | Observability Requirements | Logs (event, level, fields, when), Metrics (name, type, labels), Alerts (condition, severity, channel, runbook) |
| **api-spec** | Idempotency/Retry/Concurrency per endpoint | Idempotency key required? Retry safety? Concurrency model? Client timeout? Server timeout? |

**Example — Error Handling Matrix in feature-spec:**

```markdown
## Error Handling Matrix

ES-1: API Timeout During Import
  Trigger: POST /api/imports returns no response within 30s
  System: Circuit breaker opens; queue import for retry
  User sees: "Import is taking longer than expected. We'll notify you when it's done."
  Retry: Automatic, 3x with exponential backoff (1s, 4s, 16s)
  Rollback: If all retries fail, mark import as "failed"; no partial data committed

ES-2: Malformed CSV Row
  Trigger: Row N fails field validation
  System: Skip row, continue processing remaining rows
  User sees: "Row 5 skipped: email format invalid. 95/100 rows imported successfully."
  Retry: N/A (user can fix CSV and re-import)
  Rollback: N/A (valid rows are committed)
```

**Example — Data validation rules in data-dictionary:**

```markdown
dict.user.email:
  Type: VARCHAR(254)
  Required: Yes
  Format: RFC 5322 (loose)
  Validation: ^[^\s@]+@[^\s@]+\.[^\s@]+$
  Max length: 254 characters
  Storage: Normalized to lowercase
  Valid example: alice.smith+tag@example.com
  Invalid example: alice@.com
  On invalid: 400 Bad Request with field-level error
```

**Files to modify:**
- `.claude/skills/planning/feature-spec/SKILL.md` — add ES-#, state machine, permission matrix (~60 lines)
- `.claude/skills/planning/feature-spec/knowledge/template.md` — add template sections (~40 lines)
- `.claude/skills/data-design/data-dictionary/SKILL.md` — add validation rules per field (~20 lines)
- `.claude/skills/implementation/quality-checklist/SKILL.md` — add observability section (~20 lines)
- `.claude/skills/api-design/api-spec/SKILL.md` — add idempotency/retry/concurrency per endpoint (~15 lines)
- `.claude/skills/orchestrator/knowledge/ids.yml` — add `ES` ID prefix (~3 lines)

**Effort:** ~160 lines total

## Success Metrics

How to know if the improvements worked:

1. **"Questions the dev had to invent"** — Ask the dev at the end of each feature: how many decisions did they make that were not in the documentation? Target: < 3 per feature.

2. **Semantic dimensions covered before gate** — % of semantic dimensions covered on the first pass of ideation. Target: > 80%.

3. **Archetype probes answered** — % of archetype questions that received a concrete answer (not vague, not "I don't know"). Target: > 70%.

4. **Implementation time vs estimate** — If documentation is complete, the deviation between estimate and actual should decrease. Target: deviation < 30%.

5. **Gate iterations** — How many times a gate needs ITERATE before GO. If interviews improve, gates should pass faster. Target: average < 1.5 iterations per gate.

6. **EDGE-# resolution rate** — % of EDGE-# items that have explicit resolution (US-#, FS-#, ADR, or API error code) before Implementation Readiness Gate. Target: 100% resolved or explicitly deferred.

7. **Feedback loop yield** — Number of new archetype probes added per quarter from `/retro` learnings. Target: > 3 per quarter (framework is learning).

## New Definition of Done

A feature or product is only interview-complete when:

1. The problem is clear independently from the proposed solution
2. Archetype-specific probes have been exhausted for all applicable archetypes
3. Actors, operators, and approvers are explicit with permission matrix
4. Non-goals are explicit and traceable as NG-# IDs
5. All semantic coverage dimensions are covered or justified as not-applicable
6. Active assumptions are classified (open/deferred/assumed-default) with owners and deadlines
7. Every EDGE-# has a resolution (US-#, FS-#, ADR, API error code) or explicit deferral
8. The spec has been stress-tested via `/elicit` with zero unresolved EL-# blockers
9. Error handling matrix, state machines, validation rules, and observability are documented
10. Everything is consolidated into a single implementation packet
11. Remaining unknowns are explicit enough that engineering knows what to answer before coding
12. Gate decisions and rejected approaches are persisted for downstream phases

The fundamental shift: from **"coverage suggested by prompts"** to **"coverage enforced by state, archetypes, semantic dimensions, and developer-ready packaging"**.
