# Audit v5: ETUS PMDocs — Discovery, Brainstorm, Semantic Coverage, and Developer-Safe Handoff

**Date:** 2026-03-24  
**Framework version reviewed:** v5.3  
**Base:** Full source read across agents, skills, commands, templates, validators, hooks, state files, prior audit iterations, and external research synthesis  
**Purpose:** Define the next evolution of ETUS so it can extract product intent with much higher fidelity, detect semantic gaps earlier, and hand engineering an implementation-ready package with minimal guesswork

---

## Status of this document

This audit is intended to become the **canonic audit direction** for the ETUS improvement line.

It supersedes earlier iterations where:

- interview quality was framed mostly as "better prompts"
- coverage was still discussed mainly through counts
- handoff was treated as a documentation quantity problem
- validation was described more as policy than mechanism

This version keeps the strengths of those prior iterations, but makes the architecture more implementable.

---

## Executive Thesis

ETUS PMDocs already has the right conceptual backbone:

- ideation before specification
- upstream coverage through actors, JTBDs, journeys, use cases, edge cases, assumptions
- traceability and SST
- mode-aware workflows
- phase gates
- persistent state scaffolding

That is strong.

The main problem is not lack of methodology.

The main problem is:

> **ETUS is still strong in declared method and weak in operational enforcement.**

In practice, that means:

- interview quality still depends too much on the agent "behaving well"
- discovery can finish with surface-level completeness but deep semantic gaps
- brainstorm can still start too early or stay too generic
- validation catches structure more reliably than meaning
- engineering still reconstructs the implementation story across too many documents
- changes and rejections still propagate weakly

The ETUS evolution should therefore optimize for:

1. a stateful interview engine
2. story-based discovery
3. structured brainstorm dynamics
4. semantic coverage instead of count-only coverage
5. executable validation
6. a derived, SST-safe handoff packet
7. change propagation with rationale preservation

## Where the most important interview work actually lives

## Layer 1 — Core Interview Layer

Primary components:

- `ideate`
- `discover`
- `project-context`
- `product-vision`

This is the highest-leverage layer for extracting intent from the product owner.

This layer must:

- understand the real problem, not just the proposed solution
- extract stories from real past behavior
- expose hidden actors and operators
- surface anti-requirements and constraints
- identify hidden stakeholder conflict
- create a trustworthy problem space for downstream work

## Layer 2 — Structured Brainstorm Layer

Primary components:

- `brainstorm` inside `ideate`
- `solution-discovery`

This is the second most important interview-heavy layer, but with a different objective:

- not extract raw context
- but transform evidence into candidate directions and explicit assumptions

If discovery fails, brainstorm becomes guesswork.  
If brainstorm fails, requirements become premature fixation.

## Layer 3 — Translation Layer

Primary components:

- `feature-brief`
- `prd`
- `user-stories`
- `feature-spec`

This is not the primary interview layer, but it is where discovery and brainstorm quality must survive without collapsing into vagueness.

## Layer 4 — Assurance Layer

Primary components:

- `validate-gate`
- `check-traceability`
- `check-sst`
- future contradiction and semantic validators

This layer does not discover the idea, but must stop ambiguity and incompleteness from reaching implementation.

## Layer 5 — Change Layer

Primary components:

- `correct-course`
- workflow state
- gate decisions
- post-implementation learnings

This layer protects the system from drift after scope or direction changes.

## What already works and should be preserved

### 1. The ideation macro-sequence is sound

The order:

- ingest
- problem
- actors
- jobs
- journeys
- cases
- edges
- assumptions
- brainstorm
- synth

is correct and should remain the structural backbone.

### 2. ETUS already has seeds of anti-requirements

Signals already exist in:

- out-of-scope prompts
- constraints and guardrails
- "what must not get worse"

The gap is not absence. The gap is that these are not yet first-class, traceable, and enforced downstream.

### 3. Traceability and SST are real strengths

ETUS already has a better understanding than most frameworks of:

- authoritative document ownership
- upstream/downstream linkage
- ID-based continuity
- handoff boundaries

That is a competitive advantage and should be extended, not replaced.

### 4. Validation architecture is conceptually strong

The existing system already contains strong ideas:

- 3-layer validation
- adversarial review
- pressure tests
- assumption audit

The problem is not conceptual weakness. The problem is insufficient execution and automation.

## Refined Diagnosis

## D1 — Prompt-rich, enforcement-poor

The framework contains many good instructions:

- ask one question per message
- track unknowns
- show progress
- challenge assumptions
- present section-by-section

But these are still mostly:

- skill text
- agent instructions
- templates

and not yet:

- progression blockers
- executable gate checks
- reusable interview state
- explicit interview mechanics

This is the architectural reason interview quality still varies too much.

## D2 — Discovery is still too abstract in places

Too many prompts are still shaped like:

- What is the problem?
- Who are the users?
- What feature do you want?

These questions are useful as summaries, but weak as extraction mechanics.

The strongest pattern from research and experience is that discovery should be **story-based and past-focused**:

- tell me about the last time this happened
- walk me through it step by step
- what happened next
- what did you do to work around it
- why did that matter

Without this shift, ETUS continues to collect conceptual answers where it should collect behavioral evidence.

## D3 — Brainstorm is not yet disciplined enough as a distinct mode

Brainstorm should not start from raw notes and should not merge generation with evaluation.

It should have a clean progression:

1. synthesize themes
2. formulate opportunities / HMW prompts
3. diverge without judgment
4. cluster and shortlist
5. surface assumptions and tests

Without this structure, brainstorm risks staying too generic or becoming critique too early.

## D4 — Coverage is still too count-driven

Item counts are a useful minimum floor.

They are not enough to answer:

> Did we cover the critical dimensions of this problem type?

Examples of dimensions that can still be missing despite passing count thresholds:

- permissions and policy
- side effects and notifications
- degraded-mode behavior
- data mutations
- observability expectations
- anti-requirements
- concurrency / retry / idempotency

ETUS needs semantic coverage checks, not just item quantity checks.

## D5 — Handoff is still too distributed

Even when ETUS produces rich documentation, engineering still has to reconstruct:

- what is required
- what is forbidden
- what happens on failure
- what data changes
- who is allowed to do what
- what still remains unresolved

That is a packaging problem as much as a specification problem.

## D6 — Validation is still more policy than mechanism

The framework already says the right things:

- no placeholders
- quantified NFRs
- Given/When/Then quality
- no SST violations

But too many of these checks remain dependent on broad LLM judgment instead of explicit validation rules.

## D7 — Change propagation remains underdeveloped

ETUS is increasingly strong at generating documents from scratch.

It is still comparatively weak at:

- preserving rejected directions
- storing gate rationale
- propagating requirement changes downstream
- learning from implementation-time ambiguity

This is the biggest asymmetry left in the framework.

## Design Principles for the next version

### P1 — Separate coverage from interview execution state

Do not overload one artifact.

- `coverage-matrix.yaml` should describe semantic coverage and gate readiness
- `elicitation-state.yaml` should describe operational interview progress

### P2 — Extract through stories, not summaries

Discovery should default to behavioral narratives first and abstractions second.

### P3 — Brainstorm must be constrained by evidence

No brainstorming directly from vague problem descriptions.

### P4 — Derived handoff artifacts must never become authorities

The packet is for convenience and coordination, not for redefining truth.

### P5 — New IDs should be introduced carefully

Traceability is a strength. ID inflation can become a weakness if every concern gets its own namespace prematurely.

### P6 — Rules should be mode-aware

What is mandatory for `Product` should not automatically be mandatory for `Bug` or `Spike`.

## Recommendations

## R1 — Create `elicitation-state.yaml` as a first-class state artifact

### Why

Right now the framework can save progress, but not with enough operational specificity for interview continuity.

### What it should own

- active module
- current question
- questions asked
- answer quality
- whether escalation happened
- reflection checkpoint due
- fatigue marker
- pause/resume state
- active archetypes

### Example

```yaml
interview:
  mode: feature
  module: edges
  question_index: 3
  current_question: "What happens if the import is interrupted mid-way?"
  questions_asked: 14
  questions_remaining: 6
  answer_quality_log:
    - { q: "Who uses this?", quality: sufficient, type: confirmed }
    - { q: "How many users?", quality: insufficient, escalated: true }
  reflection_due: true
  fatigue_check_at: 15
  fatigue_offered: false
  session_paused: false
  active_archetypes: [import_export]
```

### Files

- `state_defaults.py`
- ideation templates / state templates
- sub-skills that ask questions and resume work

### Priority

P0

## R2 — Build a shared Elicitation Engine instead of duplicating interview logic

### Why

The interview behavior should not be rewritten manually inside every skill and agent.

### Responsibilities

- read `elicitation-state.yaml`
- continue where the interview stopped
- know when an answer is insufficient
- trigger vague-answer escalation
- inject reflection checkpoints
- classify "I don't know"
- manage fatigue

### Minimum rules

1. Before asking, read interview state
2. If module is already covered, summarize instead of repeating
3. If answer is vague, fire mandatory follow-up
4. After 3-4 substantive answers, trigger reflection
5. If fatigue is detected, offer continue / assume / pause

### Priority

P0

---

## R3 — Make story-based interviewing the default discovery method

### Replace abstract-first prompts with past-behavior extraction

Preferred probes:

- Tell me about the last time this happened
- Walk me through what you did step by step
- What happened after?
- What did you do to work around it?
- Why was that important?
- How much time or money did that workaround cost?

### Anti-patterns to reduce

- Would you use X?
- Is this a problem?
- Don’t you think Y matters?
- What feature do you want?

### Why this matters

Story-based extraction improves:

- detail
- context
- implicit actors
- real constraints
- hidden workarounds

### Priority

P0

---

## R4 — Add Interview Snapshots as a micro-synthesis primitive

### Why

Key detail gets lost when all raw conversation is distilled directly into large synthesis docs.

### Proposal

Add structured snapshots after each significant story:

- who told it
- context
- what happened
- memorable quote
- what broke
- workaround
- opportunities seen
- IDs fed by this story

### Important note

This does not need to become a large standalone document per interview.
It can live as a structured section inside ideation artifacts or as a lightweight structured appendix.

### Priority

P0

---

## R5 — Add archetype-aware probe packs, but start small

Start with 3 archetypes:

### Workflow / Approval

Probe for:

- states
- valid transitions
- forbidden transitions
- SLA / timeout
- delegation / escalation
- notifications by transition
- reopen rules
- audit trail

### API / Integration

Probe for:

- authentication and scopes
- retry
- idempotency
- rate limits
- partial failure
- error contract
- backward compatibility
- webhooks / replay

### Import / Export

Probe for:

- accepted formats
- max size
- preview
- invalid rows
- rollback
- resume
- duplicates
- progress
- volume
- permission to perform import/export

### Important positioning

These packs are **not** the primary extraction method.
They are a **post-story safety net** that catches what stories did not surface.

### Priority

P0

## R6 — Replace count-only coverage with semantic coverage by dimension

### Keep counts as floors, but not as decision logic

`coverage-matrix.yaml` should track dimensions such as:

- problem_clarity
- trigger_and_preconditions
- core_behavior
- success_signal
- anti_requirements
- failure_modes
- degraded_behavior
- actors_and_permissions
- data_mutations
- side_effects
- observability

Plus archetype-specific dimensions when activated.

### Mode-aware expectations

| Dimension | Product | Feature | Bug | Spike |
|---|---|---|---|---|
| problem_clarity | required | required | required | required |
| actors_and_permissions | required | required | conditional | n/a |
| success_signal | required | required | required | recommended |
| anti_requirements | required | required | recommended | n/a |
| failure_modes | required | required | required | conditional |
| degraded_behavior | recommended | conditional | conditional | n/a |
| data_mutations | conditional | conditional | conditional | n/a |
| side_effects | recommended | conditional | conditional | n/a |
| observability | recommended | conditional | recommended | n/a |

This avoids over-hardening lighter modes.

### Priority

P0

## R7 — Promote non-goals into traceable `NG-#` objects

### Why

Non-goals currently exist in prose but are too easy to lose downstream.

### Minimum fields

- statement
- reason
- adjacent behavior
- scope: permanent / deferred / conditional
- downstream docs that must respect it

### Benefit

`NG-#` helps prevent:

- scope creep
- reintroduction of rejected ideas
- implementation of "obvious" behavior that was never approved

### Priority

P0

## R8 — Make brainstorm a structured 5-stage dynamic

### Required stages

1. target opportunity
2. HMW prompts
3. divergence without judgment
4. clustering and shortlisting
5. assumptions and next tests

### Initial techniques

- HMW + clustering
- Crazy 8s
- Gut Check

### Expansion later

- reverse brainstorming
- role play
- mash-ups
- six thinking hats

### Priority

P0

## R9 — Add `/elicit` as a semantic stress-test command

### When to use it

1. after ideation, before locking feature or discovery synthesis
2. after requirements, before design
3. when the user wants to challenge what is already specified

### What it should do

It should not create new product truth.
It should interrogate existing artifacts and produce findings such as:

- ambiguity
- contradiction
- missing boundaries
- hidden assumptions
- unresolved EDGE items
- developer questions that remain unanswered

### Core checks

- Goal vs Non-Goal clash
- Actor / permission mismatch
- Missing lifecycle states
- EDGE resolution audit
- Developer simulation
- Vague quantifier scan
- Cross-doc consistency

### Priority

P0

---

## R10 — Generate a derived Implementation Packet for engineering

### Why

Engineering still has to reconstruct too much context across multiple documents.

### What the packet should contain

- context and non-goals
- actors and permissions
- business rules
- acceptance criteria
- error handling matrix
- state machine when relevant
- API contracts
- data mutations and validation
- performance requirements
- observability requirements
- implementation tasks
- open blockers

### Critical SST rule

The packet must be:

- generated
- derived
- regenerable
- non-authoritative

It consolidates authoritative sources. It does not redefine them.

### Authority map

| Packet section | Authority |
|---|---|
| Business rules | feature-spec / feature-brief |
| Acceptance criteria | user-stories |
| API contracts | api-spec |
| Data semantics and validation | data-dictionary + database-spec |
| NFRs | tech-spec |
| Permissions | feature-spec or tech-spec |
| Observability | upstream observability requirements, then checked by quality-checklist |

### Important refinement

Do **not** introduce a new ID namespace for error scenarios yet.

For the first implementation, model error handling as a required structured matrix inside `feature-spec`, tied to existing IDs:

- `EDGE-#`
- `US-#`
- `FS-#`
- API error codes

Only create a new namespace later if there is evidence that traceability truly needs it.

### Priority

P0

---

## R11 — Move observability authority upstream

### Why

Observability should not first appear as a release-readiness checklist item.

If it matters for implementation, it needs to exist upstream as a requirement.

### Recommendation

Make observability authoritative in one of these ways:

- as part of `tech-spec`
- or as a dedicated upstream observability requirements section referenced by the packet

Then use `quality-checklist` to validate implementation readiness against that requirement.

### Priority

P1

---

## R12 — Add executable validators for placeholders, vagueness, and semantic omissions

### Strong candidates for executable checks

- bracket placeholders in required sections
- empty required table cells
- NFR without number + unit
- repeated vague terms without metrics
- required sections missing or structurally empty

### More advanced validators

- no error-path sibling for high-risk happy-path flow
- permission claim without formal role definition
- contradiction between DB constraints and data dictionary
- contradiction between API contract and feature rules

### Important caveat

Avoid simplistic regex-only enforcement that causes too many false positives in:

- examples
- templates
- quoted guidance

### Priority

P0

---

## R13 — Persist gate decisions with rationale and rejected approaches

Current gate state tracks status too lightly.

It should also persist:

- feedback
- rationale
- rejected approaches
- unresolved objections
- iteration count

This should feed:

- workflow state
- next prompts
- future iterations
- post-change propagation

### Priority

P1

---

## R14 — Strengthen correct-course into actual change propagation

### Needed behavior

When a requirement changes:

1. detect impacted docs
2. map affected IDs
3. propose downstream edits
4. preserve rationale for change
5. update feature and product views consistently

### Why

Without this, ETUS is strong at generation but weak at controlled evolution.

### Priority

P1

---

## R15 — Add a post-implementation learning loop

After implementation, ask:

- what did the dev still need to decide alone?
- what should have been asked earlier?
- which archetype or dimension missed it?

Store this in learnings and feed it back into:

- archetype probe packs
- elicitation engine
- semantic validators

### Priority

P1 late / P2

## Definition of success

ETUS should consider a work item interview-complete only when:

1. the problem is stated independently from the proposed solution
2. the main actor, operator, and approver are explicit
3. anti-requirements are explicit and traceable
4. critical journeys and failure paths are covered
5. unresolved assumptions are classified and owned
6. discovery evidence has been translated into disciplined brainstorm directions
7. every important `EDGE-#` has a downstream resolution or explicit deferral
8. the implementation packet can be generated without inventing new truth
9. observability, error handling, validation rules, and permissions are present in upstream authoritative sources

---

## Final conclusion

The next leap for ETUS is not adding more documentation volume.

It is making the framework more:

- stateful in interviews
- behavioral in discovery
- disciplined in brainstorm
- semantic in coverage
- mechanical in validation
- safe in developer handoff
- resilient under change

If ETUS gets those seven things right, it stops being only a strong documentation framework and becomes a real operating system for elicitation, decision-making, and implementation readiness.
