# Bug/Hotfix Mode

Document a bug fix with root cause analysis, fix plan, and test plan. Generates a single tech-spec document.

## Usage
```
/bugfix              -> Start bug documentation
/bugfix [description] -> Start with bug description
```

## Workflow

1. Detect or confirm Bug mode
2. Run `/ideate` first in failure-understanding mode
3. Interview: what broke, when it started, impact severity, reproduction steps
4. Generate single tech-spec-{slug}.md with:
   - **Problem description** — What is broken, symptoms, affected users
   - **Root cause analysis** — Why it broke, contributing factors
   - **Affected components** — Which services, tables, endpoints, screens are involved
   - **Fix approach** — Step-by-step plan to resolve the issue
   - **Test plan** — How to verify the fix works and nothing else broke
   - **Rollback plan** — How to revert if the fix causes new issues
5. Save to docs/ets/projects/{project-slug}/bugs/

## Output

```
docs/ets/projects/{project-slug}/bugs/
└── tech-spec-{slug}.md    # Single document with all sections above
```

## Dependencies

- `opportunity-pack.md` — BLOCKS (generated by `/ideate`, mandatory)
- `coverage-matrix.yaml` — BLOCKS (generated by `/ideate`, mandatory)
- No upstream documents required.
- Optionally references existing US-# or PRD-F-# if the bug relates to a documented feature.

## Validation

- Problem described (non-empty)
- Fix approach present and actionable
- Test plan present with at least 1 test case
- No quality gates.

## Agents

- None by default (orchestrator handles directly)
- For complex bugs affecting multiple subsystems, optionally spawns implementation-agent
